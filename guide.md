# 강점 분석 가이드

> **"어떤 데이터로, 어떤 원칙에 따라"** 개인의 강점을 직무·공헌으로 연결하는지를 정리한 방법론 문서다.
> 프로젝트가 바뀌어도 이 절차는 그대로 쓴다 — 데이터 소스만 갈아 끼운다.

---

## 1. 목적

강점 **진단에서 멈추지 않는다.** `강점 → 직무 → 공헌(측정 가능한 성과)`까지 잇는 게 목표다.

- **이론(원칙)**: [`docs/strength-analysis.md`](docs/strength-analysis.md) — 드러커 "강점 활용" 원칙
- **성과 정의**: [`docs/frontend-strength.md`](docs/frontend-strength.md) — "무엇을 했다"가 아니라 "어떤 숫자를 움직였다"가 성과
- **절차(이 문서)**: 위 원칙을 실제 데이터에 적용하는 방법

---

## 2. 이론적 토대 — `strength-analysis.md`

배치의 모든 판단은 이 원칙에서 나온다:

1. **직무를 사람에게 맞춘다** — 빈자리에 사람을 채우지(staffing the slot) 않고, 강점에 책임을 모은다.
2. **강점은 공헌으로 끝나야 한다** — 검증 사슬 `강점 → 산출물 → 조직 기여 → 측정 지표`. 사슬이 끊기면 성과로 번역 안 된 것.
3. **성과는 N분의 1로 안 쪼갠다** — 성과마다 **단독 오너 1명**. 여럿이 나누면 끝까지 책임질 사람이 사라진다.
4. **약점은 사람이 아니라 구조가 무효화한다** — 체크리스트·페어리뷰·오너십 분리·템플릿.
5. **실적으로만 평가한다** — 잠재력(가능성)이 아니라 **드러난 성과**를, 구체적 기대치와 비교해.

---

## 3. 데이터 3소스 (삼각측량)

| 소스                         | 무엇을 말하나                          | 한계 (오독 주의)                                             | 위치                   |
| ---------------------------- | -------------------------------------- | ------------------------------------------------------------ | ---------------------- |
| **갤럽 CliftonStrengths 34** | 타고난 **재능·동기** — 강점의 1차 신호 | 재능 ≠ 스킬·성과. 자기응답 기반                              | (외부 PDF · 개인정보)  |
| **정기 면담**                | **동기·몰입·회피·자기인식**            | 자기보고 편향, 시점 의존. 자기인식이 *학습된 방어*일 수 있음 | `meeting/`             |
| **업무 이력**                | 실제 **활동·도메인 친숙도**            | 건수 = 활동량 **≠ 성과**(미측정). 램프업 비용 신호일 뿐      | `workboard/`           |
| _(있으면)_ **실적 지표(KR)** | 드러난 **성과**                        | 아직 미측정이면 N/A → 잠정 배치 단계                         | `frontend-scoreboard/` |

> **핵심 원칙 — 단일 소스로 결론짓지 않는다.**
> **세 소스가 같은 곳을 가리킬 때만** 강점으로 확정한다(triangulation).
> 소스가 서로 충돌하면 → 숨기지 말고 **플래그**를 달아 재평가 대상으로 남긴다.

---

## 4. 분석 절차 (6단계)

1. **수집** — 3소스를 모은다 (갤럽 없으면 그 사람은 면담+업무 2소스로, 단 신뢰도 낮음을 명시).
2. **교차(triangulation)** — `갤럽 테마 ↔ 실제 업무 증거 ↔ 면담 발언`을 한 줄씩 대응시키는 표를 만든다.
3. **강점/약점 도출** — 도메인 분포·상위/하위 테마를 본다. 상위에 몰린 도메인이 엔진, 하위에 몰린 도메인이 약점.
4. **직무를 강점에 맞춘다** — 강점 축에 책임을 모으고 **단독 오너**를 둔다. 강점이 안 맞는 일은 빼거나 분리한다.
5. **약점은 구조로 무효화** — 약점 영역의 일은 페어·오너십 분리·체크리스트로 덮는다(사람 탓 X).
6. **잠정 배치 → 실적으로 재평가** — 활동량으로 우열 금지. **재평가 기한을 명시**하고 KR(실적)로 확정한다.

---

## 5. 소스별 해석 주의 (함정)

- **갤럽** = 재능이지 성취가 아니다. 약한 도메인도 노력·스킬로 수행할 수 있다. 단, _강점 기반 배치_ 문서에서는 재능이 **1차 신호**다.
- **면담** = 자기인식이 *학습된 방어(coping)*일 수 있다. 예: "안정적인 일을 원한다"류의 진술이 실제 재능이 아니라 압박 회피의 결과일 수 있음 → 갤럽과 대조해 분별한다.
- **업무 이력** = 건수·도메인은 **친숙도(램프업 비용)**일 뿐 성과가 아니다. "많이 했다"로 우열을 매기지 않는다.
- **상충할 때** = 더 강한 신호로 덮어쓰지 말고 **플래그**로 드러낸다. 그게 다음 재평가의 입력이 된다.

## 7. 1인 분석 체크리스트

- [ ] 3소스를 다 모았나 (없으면 신뢰도 한계를 명시했나)
- [ ] 교차(triangulation) 표를 만들었나
- [ ] 강점 → 직무 → **공헌(측정 지표)**까지 사슬이 이어지나
- [ ] 약점 영역을 **구조**로 덮었나 (사람 탓 X)
- [ ] 성과마다 **단독 오너**인가 (N분의 1 아님)
- [ ] 활동량(건수)으로 우열을 매기지 않았나
- [ ] 소스 충돌을 플래그로 남겼나
- [ ] **재평가 기한**을 명시했나
