# 강점을 '팀 성과'로 연결하기 — 실전 적용 노트

> 위 드러커 원칙을 한 팀(파트) 단위로 옮길 때의 보편 원칙. 개인이 아니라 **여러 명을 어떻게 배치하고 성과를 어떻게 나눌 것인가**에 대한 정리.

#### 1. 강점은 '공헌(contribution)'으로 끝나야 한다

- 강점 진단에서 멈추면 안 된다. **강점 → 직무 → 조직 공헌(성과)**까지 이어져야 비로소 '설계'다.
- 드러커의 진짜 질문은 "그가 무엇을 잘하는가"가 아니라 **"그가 어떤 공헌을 하는가"**이다.
- 검증 사슬: `강점 → 산출물 → 조직 기여 → 측정 가능한 지표`. 이 사슬이 끊기는 강점은 성과로 번역되지 않은 것이다.

#### 2. 성과는 'N분의 1'로 쪼개지 않는다

- 인원이 여럿이라고 **같은 성과를 똑같이 나누는 것**은 드러커가 비판한 *"빈자리를 채우는(staffing the slot)"* 사고다.
- 하나의 성과를 여럿이 나눠 가지면, 그 숫자를 **끝까지 책임질 단독 오너가 사라진다.** 모두가 "두루 하는 사람"이 되어 버린다.
- 성과는 분할 노동이 아니라 **소유(ownership)**다.

#### 3. 강점별로 배분하고, 약점은 서로 무효화한다

- 성과마다 **단독 책임자(accountable owner) 1명** — 그 성과에 강점이 맞는 사람.
- 오너십 ≠ 혼자 다 함. **책임지고 끌고 가는 사람**이 한 명이라는 뜻이고, 실행은 함께 한다.
- 팀원들의 강점 축이 **서로 겹치지 않게** 배치한다. 그러면 한 사람의 약점을 다른 사람의 강점이 덮는다 → *"조직은 약점을 무의미하게 만드는 도구"*가 실현된다.
- 반대로 같은 일을 여럿이 나누면, 강점뿐 아니라 **약점도 그대로 인원수만큼 복제된다.**

#### 4. '개인이 소유하는 성과' vs '팀 공통 역량'을 구분한다

| 유형 | 처리 방식 | 성격 |
| --- | --- | --- |
| **개인이 소유하는 성과** | 강점 맞는 1인에게 오너십 부여 | 축별로 한 명이 책임 (예: 검색 유입 / 성능 / 품질) |
| **팀 공통 역량** | 쪼개지 않고 셋 다 체화하는 문화·표준 | 모두의 기본기 (예: AI 활용으로 공수↓·정확도↑, 코드 품질) |

→ 어떤 성과는 한 명이 소유하고, 어떤 역량은 팀 전체가 함께 가진다. 이 둘을 섞으면 안 된다.

#### 5. 팀의 목표와 기여를 먼저 정의한다

개인 강점은 **팀 목표에 연결될 때만** 성과가 된다. 그래서 개인 배치 전에 "팀이 회사에 무엇을 기여하는가(기여 축)"를 먼저 정의해야 한다.

- **팀 미션 (예시 골격):** 웹 제품의 *검색 유입 · 속도 · 품질*로 회사의 *성장과 효율*에 기여한다 — 적은 인원으로, AI를 레버리지 삼아.
- **기여 축(= 팀 목표)과 소유 형태:**
  - 검색 유입 성장(SEO) → organic 트래픽 = 신규 유저 획득(마케팅비 대체) — *[개인 소유]*
  - AI 레버리지 → 공수 단축 + 정확도 증대 — *[팀 공통 역량]*
  - 웹 성능 · 안정성(Core Web Vitals) → 검색 랭킹·이탈률·인프라 비용 — *[개인 소유]*
  - 협업 효율 · 품질 → 기획-개발 의사소통/QA 공수↓, 기획→배포 리드타임↓ — *[개인 소유]*
- 각 축에는 **측정 지표**를 붙인다(예: organic 트래픽, LCP/INP, 재작업률, QA 반려율, 리드타임). 지표 없는 목표는 공헌으로 검증되지 않는다.

#### 6. 직무 설계 원칙 (요약)

1. **높고 여유 있게 설계한다** — 좁고 쉬운 일은 능력을 시험할 수 없게 만든다. 도전적인 범위로 둔다.
2. **직무를 사람에게 맞춘다** — "자연의 섭리·하느님의 손이 만든 직무"는 없다. 강점에 맞춰 책임을 모은다.
3. **약점은 구조가 무효화한다** — 사람을 탓하지 않고 체크리스트·페어리뷰·템플릿으로 빈틈을 메운다.
4. **실적으로만 평가한다** — 잠재력(가능성)이 아니라 드러난 성과를, 구체적 기대치와 비교해 평가한다.
