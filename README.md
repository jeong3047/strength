# 강점 기반 배치 · 성과 정의 방법론

개인의 **강점**을 **직무 배치**와 **측정 가능한 성과**로 잇기 위한 방법론 모음입니다.
특정 팀·개인 데이터(면담·평가·내부 전략)는 포함하지 않으며, **재사용 가능한 절차와 원칙**만 공개합니다.

> 이 저장소는 "어떤 데이터로, 어떤 원칙에 따라" 강점을 공헌으로 연결하는지를 다룹니다.
> 프로젝트가 바뀌어도 절차는 그대로 두고 **데이터 소스만 갈아 끼웁니다.**

## 🎯 핵심 원칙

1. **강점 → 직무 → 공헌**: 강점 진단에서 멈추지 않고 측정 가능한 성과까지 연결한다.
2. **교차검증**: `재능 신호(갤럽) ↔ 실제 업무 증거 ↔ 자기보고(면담)`를 한 줄씩 대응시켜 분별한다.
3. **활동량 ≠ 성과**: 작업 "건수"는 도메인 친숙도 신호일 뿐, 숫자를 움직인 것만 성과로 본다.
4. **잠정 배치 → 계측 → 재평가**: 강점 기반 배치는 확정이 아니라 가설이며, 계측 후 실적으로 재배치한다.

## 🚀 빠른 시작 — 소스 채우고 한 명령으로 보고서 생성

처음 보는 사람도 **소스만 채우면**, 명령 한 번으로 **조직 보고서 + 파트원별 세부 보고서**가 HTML로 한꺼번에 만들어진다.

### 1단계 — 소스 채우기

| 소스 | 위치 | 필수 | 내용 |
| --- | --- | --- | --- |
| **면담 기록** (1:1 노트) | `meeting/*-meeting.md` | ✅ | 동기·몰입·회피·자기인식 |
| **업무 이력** (작업 로그/PR/티켓) | `workboard/*.csv` | ✅ | 활동·도메인 친숙도 |
| **갤럽 CliftonStrengths 34** | 외부 PDF/텍스트 | ⬜ **선택** | 있으면 재능 1차 신호, **없으면 2소스로 진행** |
| 실적 지표(KR) | 있으면 | ⬜ 선택 | 드러난 성과(숫자). 미측정이면 빈칸 |

> 폴더별로 무엇을 넣는지는 각 폴더의 README(`meeting/`·`workboard/`·`frontend-scoreboard/`) 참고.
> 갤럽이 없으면 면담+업무이력만으로 돌아가며, 보고서에 **"잠정 신뢰도"**가 표기된다.

### 2단계 — 명령 실행

- **Claude Code**: `/strength-report`
- **그 외 LLM**: [`report-spec.md`](report-spec.md) + [`guide.md`](guide.md)를 프롬프트로 넣고 "소스를 읽어 스펙대로 HTML 생성"을 지시.

명령은 내부적으로 [`guide.md`](guide.md) 절차(`갤럽 ↔ 업무증거 ↔ 면담` 교차검증 → 강점 도출 → 직무 배치 → 단독 오너 → 재평가 플래그)를 그대로 수행한다.

### 3단계 — 결과 (한 번에 생성)

`frontend-scoreboard/`에 다음이 만들어진다:

```
frontend-scoreboard/
├── index.html          ← 조직 보고서 (전략·노스스타·KR 롤업·월간 트래킹)
├── member-<id1>.html   ← 파트원 세부 (강점 카드·현황·담당 KR)
├── member-<id2>.html
└── …                    (면담이 있는 인원 수만큼)
```

> ⚠️ 생성물은 실명·개인 분석을 포함하므로 `frontend-scoreboard/`는 **로컬 전용**(`.gitignore`)이다. 커밋되지 않는다.
> 출력 구조·섹션·데이터 무결성 규칙의 전체 정의는 [`report-spec.md`](report-spec.md) 참고.
